# DATA & ASSERTIONS — CORE

## `pytest.approx` for Numeric Comparisons

[ref: #data-assertions-pytest-approx]

Use `pytest.approx` for all floating-point and numerical comparisons.
Never assert exact equality on computed floats.
The default relative tolerance is `1e-6`, and an implicit absolute tolerance of `1e-12` handles values near zero unless you set `abs` explicitly.
Relative tolerance is computed with respect to the *expected* value.
Provide `abs=` when comparing values close to zero, and use `nan_ok=True` to treat NaN as equal to NaN.
`pytest.approx` supports numbers, sequences, dict values, and NumPy arrays recursively.

Scalar comparison should verify a structural invariant rather than repeating the production formula:

```python
import math

import pytest
from dataclasses import dataclass
from faker import Faker


@dataclass(frozen=True)
class Circle:
    radius: float

    @property
    def area(self) -> float:
        return math.pi * self.radius ** 2


def test_circle_area_is_pi_times_radius_squared(fake: Faker) -> None:
    """
    Given: a circle with a random positive radius.
    When: its area is computed.
    Then: area divided by radius squared approximates pi.
    """
    # --- Arrange ---
    radius = fake.pyfloat(min_value=0.1, max_value=100.0, positive=True)
    circle = Circle(radius=radius)

    # --- Act ---
    result = circle.area

    # --- Assert ---
    assert result / (radius ** 2) == pytest.approx(math.pi, rel=1e-5)
```

Near-zero values need an explicit absolute tolerance:

```python
import pytest
from faker import Faker


def test_tiny_offset_is_within_absolute_tolerance(fake: Faker) -> None:
    """
    Given: a tiny positive base and a small relative offset.
    When: the offset is added to the base.
    Then: the sum is within the configured absolute tolerance of the base.
    """
    # --- Arrange ---
    base = fake.pyfloat(min_value=1e-12, max_value=1e-9, positive=True)
    offset = base * fake.pyfloat(min_value=1e-4, max_value=1e-3, positive=True)
    expected = base

    # --- Act ---
    result = base + offset

    # --- Assert ---
    assert result == pytest.approx(expected, abs=base * 1e-2)
```

Collections and dicts are compared recursively:

```python
import pytest
from faker import Faker


def test_batch_measurements_match(fake: Faker) -> None:
    """
    Given: a dict of random measurements and a tiny scale factor.
    When: each measurement is scaled.
    Then: the scaled values match the originals within relative tolerance.
    """
    # --- Arrange ---
    count = fake.pyint(min_value=2, max_value=5)
    keys = [fake.unique.word() for _ in range(count)]
    expected = {k: fake.pyfloat(min_value=1.0, max_value=100.0, positive=True) for k in keys}
    factor = 1 + fake.pyfloat(min_value=1e-7, max_value=1e-6, positive=True)

    # --- Act ---
    result = {k: v * factor for k, v in expected.items()}

    # --- Assert ---
    assert result == pytest.approx(expected, rel=1e-5)
```

Use `nan_ok=True` when NaN is a valid value:

```python
import math

import pytest
from faker import Faker


def test_sensor_readings_treat_nan_as_equal(fake: Faker) -> None:
    """
    Given: a list containing a base value, NaN, and a near-equal value.
    When: compared with pytest.approx and nan_ok=True.
    Then: NaN equals NaN and the near-equal value matches within tolerance.
    """
    # --- Arrange ---
    base = fake.pyfloat(min_value=1.0, max_value=100.0, positive=True)
    result = [base, math.nan, base * 1.000001]
    expected = [base, math.nan, base]

    # --- Assert ---
    assert result == pytest.approx(expected, rel=1e-5, nan_ok=True)
```

**Variety booster:** Parametrize the magnitude (large vs. near-zero), the collection size, and the perturbation scale so a single assertion body covers scalar, sequence, and dict cases.

## `__repr__` for Debuggability

[ref: #data-assertions-repr]

Test data classes SHOULD implement `__repr__` so that `assert user1 == user2` failures produce human-readable diffs.
The `@dataclass` decorator generates a sensible `__repr__` automatically.
Do NOT write dedicated tests for `__repr__` string format — it is a developer-experience aid, not an assertion target.

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class User:
    id: str
    email: str
    role: str
    is_active: bool
```

When a test asserts `assert actual_user == expected_user`, pytest prints the `__repr__` of both objects if they differ.
A good `__repr__` turns that diff into a one-line diagnosis instead of a memory address.

**Variety booster:** Assert whole-object equality across several boundary scenarios (default, minimal, and fully populated values) and let pytest print the generated `__repr__` diff for each failure, surfacing formatting issues through real assertions rather than snapshot tests.

## Custom Assertion Comparison (`__eq__`)

[ref: #data-assertions-eq]

If `assert a == b` fails with an unreadable diff, implement `__eq__` (and `__repr__`) on the data class.
The simplest reliable way is `@dataclass`, which generates value-based equality.
Do NOT chain manual field assertions.

Bad — brittle and verbose:

```python
def test_users_match_brittle(actual: User, expected: User) -> None:
    """
    Given: two user objects.
    When: their fields are compared manually.
    Then: the brittle chained assertions pass.
    """
    # --- Assert ---
    assert actual.id == expected.id
    assert actual.email == expected.email
    assert actual.role == expected.role
    assert actual.is_active == expected.is_active
```

Good — relies on well-defined equality:

```python
def test_users_match_clean(actual: User, expected: User) -> None:
    """
    Given: two user objects with value-based equality.
    When: they are compared directly.
    Then: the equality assertion passes.
    """
    # --- Assert ---
    assert actual == expected
```

**Variety booster:** Once `__eq__` is defined, use `@pytest.mark.parametrize` to compare the same object under many small mutations (role change, active toggle, email update) with a single `assert actual == expected` body.

## Test Data Builders vs Object Mothers

[ref: #data-assertions-builders-mothers]

A **builder** is a flexible factory with sensible defaults and override kwargs.
Use it when a test needs fine-grained control over a subset of fields.
An **object mother** is a fixed, named helper for a common scenario.
Use it when the scenario itself is semantically meaningful.
A hybrid approach — a mother that returns a configured builder — combines convenience and flexibility.

```python
from dataclasses import dataclass
from enum import StrEnum

from faker import Faker


class Role(StrEnum):
    MEMBER = "member"
    ADMIN = "admin"


@dataclass(frozen=True)
class User:
    id: str
    email: str
    role: Role
    is_active: bool


class UserBuilder:
    def __init__(self, fake: Faker) -> None:
        self._fake = fake
        self._role = Role.MEMBER
        self._is_active = fake.boolean()

    def with_role(self, role: Role) -> "UserBuilder":
        self._role = role
        return self

    def with_active(self, is_active: bool) -> "UserBuilder":
        self._is_active = is_active
        return self

    def build(self) -> User:
        return User(
            id=self._fake.uuid4(),
            email=self._fake.fake_email(),
            role=self._role,
            is_active=self._is_active,
        )


# Object mothers returning a configured builder
def active_admin(fake: Faker) -> User:
    return UserBuilder(fake).with_role(Role.ADMIN).with_active(True).build()


def inactive_member(fake: Faker) -> User:
    return UserBuilder(fake).with_role(Role.MEMBER).with_active(False).build()
```

Usage in tests:

```python
def test_admin_can_access_dashboard(fake: Faker) -> None:
    """
    Given: an active admin user.
    When: dashboard visibility is checked.
    Then: the admin can see the dashboard.
    """
    # --- Arrange ---
    admin = active_admin(fake)

    # --- Act ---
    result = dashboard.is_visible_to(admin)

    # --- Assert ---
    assert result is True


def test_default_user_role_is_member(fake: Faker) -> None:
    """
    Given: a user built with default builder settings.
    When: the user's role is inspected.
    Then: the default role is member.
    """
    # --- Arrange ---
    user = UserBuilder(fake).build()

    # --- Act ---
    role = user.role

    # --- Assert ---
    assert role == Role.MEMBER
```

**Variety booster:** Define mothers as thin wrappers around the builder, then drive them through a shared parametrized test that receives the mother function as a parameter, covering every scenario with one assertion.

## Test Functions MUST Return `None`

[ref: #data-assertions-return-none]

A non-`None` return value from a test **function** triggers a pytest warning and usually signals a logic error, such as accidentally returning a value instead of asserting.
Fixtures that use `yield` are exempt because they return generators, which is pytest's expected pattern.
Every test function must therefore be annotated to return `None` and must not return any other value.

Wrong — the test returns instead of asserting:

```python
def test_user_count_returns_int(fake: Faker, users: list[User]) -> None:
    """
    Given: a non-empty list of users.
    When: the test returns the count instead of asserting.
    Then: the anti-pattern is visible in the test body.
    """
    # --- Arrange ---
    # users fixture is provided.

    # --- Act ---
    count = len(users)

    # --- Assert ---
    # logic error: test should assert, not return.
    return count
```

Correct — the test asserts the invariant:

```python
def test_user_count_is_positive(fake: Faker, users: list[User]) -> None:
    """
    Given: a non-empty list of users.
    When: the list length is checked.
    Then: the count is positive.
    """
    # --- Arrange ---
    # users fixture is provided.

    # --- Act ---
    count = len(users)

    # --- Assert ---
    assert count > 0
```

**Variety booster:** Parametrize the candidate value returned by a helper across categories (string, integer, object), assert on its type or truthiness in every case, and keep the test body to a single assertion.
